Floating Lyrics 專案架構與資料庫報告

這份報告包含了目前專案的 實體關聯圖 (ER Diagram) 以及 系統流程圖 (System Flow Chart)。

1. 資料庫實體關聯圖 (ER Diagram)

目前專案使用的是 SQLite 資料庫（檔案名稱為 lyrics_data.db），共有四張主要的資料表，負責快取歌詞、假名修正、時間軸微調，以及使用者的播放歷史紀錄。

資料表說明

  • cache (快取歌詞)：儲存從網路上抓取下來的歌詞，避免重複發送請求。複合主鍵為 (artist, title)。
  • word_corrections (假名修正)：儲存使用者自訂的日文漢字讀音修正，確保下次播放同一首歌時，該漢字會直接套用正確的假名。
  • sync_offsets (時間軸微調)：儲存使用者對特定歌曲調整的時間軸偏移量。
  • listening_history (播放歷史)：記錄使用者的播放次數與時間，作為網頁端「統計數據 (Stats)」與「排行榜」的資料來源。

2. 系統架構與資料流程圖 (System Flow Chart)

流程亮點說明

  1. 即時狀態監控 (media_monitor.py)：透過 Windows API 不斷獲取 Spotify 等播放器的狀態（歌曲、歌手、進度），並以 JSON 格式即時輸出給 Node 伺服器 (server.js)。
  2. 網頁與桌面雙核同步：
    • server.js 作為資料中樞，為網頁前端 (app.js) 提供 RESTful APIs 進行歌詞同步、圖表渲染。
    • main.py 則作為獨立的輕量級 PyQt6 桌面懸浮視窗，它們共用同一個 lyrics_data.db，確保無論在哪一端修改「時間微調」或「假名」，另一端都能即時獲取。
  3. 資料持久化：每次歌曲切換時，Node 伺服器會將該次播放紀錄寫入 SQLite 資料庫中，作為使用者個人聽歌習慣的大數據來源。
